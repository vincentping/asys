# ASys 白皮书
## Agentic System Interface

**ASys**（Agentic System Interface，智能体系统接口）是一个探索性协议，尝试为 AI Agent 提供一条二进制原生的系统操控通道——通过强类型 APDU 二进制帧直接与系统交互，而不是通过 Shell 文本。协议对所有 POSIX 兼容系统开放，参考实现 asyd 以 Linux 为首发平台。

守护进程 **asyd**（ASys Daemon）运行于节点侧，负责接受、鉴权并执行 ASys 指令。

> 协议版本：v1.0（`0x0100`）| 参考实现：asyd v0.3.1 | 2026-06-01
>
> 协议版本与软件版本独立管理。v1.0 覆盖已稳定实现的部分；规范中已定义但尚未实现的功能标注为 *planned*。

## 1. 背景与出发点

### 1.1 现有工具的设计前提

Unix Shell（1971）和 POSIX（1988）的设计初衷是**为人服务**：命令输出对人眼友好、交互以秒为单位、权限模型以用户身份为中心。几十年来，这套体系运转良好，并衍生出 SSH、Ansible、K8s Operator 等一系列成熟工具，在各自的场景下经过了大规模生产验证。

当操作者从人类工程师变为 AI Agent 时，有几个具体问题值得重新思考：

**文本解析的开销与脆弱性**：Agent 通过 SSH 执行命令，拿到的是非结构化文本输出——需要用正则或另一个 LLM 解析。格式的微小变化（如 `ps` 输出的列宽变化）可能导致解析失败。这不是不能解决，但每次都要处理。

**权限粒度与 Agent 的失控风险**：SSH 的权限模型是会话级的，给 Agent 一个 SSH 会话，实际上给的是相当宽泛的操作空间。在 Agent 可能产生幻觉的前提下，这个空间比人类使用时风险更高。

**极端场景下的可用性**：在 Out-Of-Memory/OOM 或磁盘满的情况下，SSH 登录可能超时，Ansible 可能因无法分配内存而启动失败。如果希望 Agent 在这种时刻还能做一些诊断或有限干预，现有工具的可用性是个问题。

**采样频率与连接开销**：Agent 如果需要高频采样系统状态（比如每秒多次），SSH 每次连接的握手开销会成为瓶颈。长连接方案可以解决，但现有工具没有专门为这个场景优化。

这些问题有些可以通过在现有工具上加一层来缓解，有些则是工具设计前提带来的结构性约束。ASys 是在这个背景下的一次尝试：如果专门为 AI Agent 设计一个系统接口，从第一性原理出发，它应该长什么样。

### 1.2 设计出发点：Agent 需要什么样的接口

从 Agent 的几个实际特征出发，可以推导出一些接口的设计偏好：

**Agent 处理结构化数据比解析文本更可靠。** 固定字节偏移的二进制格式，解析成本趋近于零，也不存在格式漂移的问题。这是 ASys 选择 APDU 二进制帧的出发点。

**长连接比每次重新握手更适合高频交互。** 一旦连接建立，后续指令的往返延迟接近物理极限。对于需要高频采样或快速连续操作的场景，避免重复握手的开销有实际意义。

**指令级权限比会话级权限更适合 Agent。** Agent 可能犯错，且错误是高速的。如果权限边界足够精细——只有明确授权的指令才存在，未授权的在协议层就被拒掉——那么 Agent 幻觉的破坏半径是可以事先界定的。

**可审计性对于 AI Agent 特别重要。** 人类运维的决策过程通常是可观察的，AI Agent 的决策过程往往不透明。如果接口层内建审计，每次系统干预都有完整记录，人类在事后可以审查 Agent 的行为，这是建立信任的基础。

这四个出发点对应了 ASys 的核心设计：强类型 APDU 二进制帧、Noise IK 加密长连接、基于 Capability Map 的正向授权、Epoch_ID 防重放与 append-only 审计链。每个选择都有其权衡，设计注记（`asys-design-notes.md`）里有详细推导。

## 2. 设计哲学

### 2.1 原则一：确定性优先（Determinism First）

每一条 ASys 指令对应且仅对应一个原子操作。Agent 发出 `SVC_RESTART`，得到的永远是"重启服务"这一个结果，不受环境、Shell 版本、系统状态的影响。确定性是 ASys 协议合法性的基础。

**二进制是永恒的（Binary Immortality）**

ASys 协议定义的 `0x03` 指令返回的永远是固定的字节布局。无论底层是 RHEL 9.x 还是 10.x，Agent 拿到的 PID 永远在第 2 字节，CPU 占比永远在第 6 字节。固定的字节布局意味着 Agent 的解析逻辑可以是一行 `struct.unpack`，而不是一段随时可能失效的正则表达式。这不只是性能问题，而是**可靠性的量级差异**。

**结论：算力完全用于决策**

当解析的负担降为零，Agent 不再需要消耗任何 Token 去猜测文本含义，也无需维护额外的幻觉验证逻辑。100% 的计算量用于真正重要的事：感知、推理、决策。对于 Agent 架构师而言，这意味着更低的推理成本和更简洁的错误处理路径。

### 2.2 原则二：消除物理攻击面（Instruction Existence over Access Control）

Agent 必须持有对应 Capability 才能调用特定指令。权限不是会话级的"全或无"，而是**指令级的精确映射**。

**正向授权 vs 负向拦截**

传统安全模型（SSH/sudo）是为人类操作员设计的——人的操作是探索性的，意图难以预先声明，"宽泛默认权限 + **负向拦截**"符合人类的工作方式，是合理的设计选择。但当操作主体变为 AI Agent 时，这个假设不再成立：Agent 的行为是程序化的，权限边界完全可以在部署时精确声明。

ASys 采用**正向授权**的逻辑：系统对 Agent 是一片漆黑，只有在 Capability Map 中显式点亮的指令才是"物理存在"的。权限不是独立的 ACL 规则，而是直接编码在 Capability Map 中——未在位图中注册的指令在解包阶段直接丢弃，不进入任何执行流。

**结论**

最小权限不是 ASys 的安全策略选项，而是其协议的物理属性。没有注册，即没有存在；没有存在，即没有攻击面。

### 2.3 原则三：机制与策略分离（Separation of Mechanism and Policy）

Core ISA（`0x00-0x0F`）已定义指令永久锁定，一旦发布即不可更改。Standard ISA（`0x20-0x8F`）已定义指令的语义同样锁定，但空槽可由协议组按需添加新指令。Protocol Control（`0x10-0x1F`）由协议组保留用于内部控制。`0x90-0xBF` 为 Standard ISA 预留区间，供协议组未来新增标准指令组使用。Vendor Extensions（`0xC0-0xFF`）由各领域管理员自由定义，ASys 只提供鉴权与传输的"刀柄"，不干涉领域逻辑的"刀刃"。

**硬内核，软边界**

- **Core ISA（协议宪法，不可变）**：它是 Agent 观察世界的"视网膜"。`0x02` 永远返回系统负载，无论底层是 Linux、BSD 还是 50 年后的某个 POSIX 兼容系统。这种跨代兼容性是任何 Agent 敢于直接调用 ASys 指令的底气。Core ISA 降低了接入成本（Low Entry Barrier）——任何 Agent 无需了解节点细节，握手即可感知。

- **Standard ISA + Vendor Extensions（领域法律，可演化）**：它是不同垂直领域的"手术刀"。Standard ISA（`0x20-0x8F`）由协议组定义覆盖通用场景；Vendor Extensions（`0xC0-0xFF`）则由领域专家自由扩展——数据库 Agent 需要 `PROC_CHECKPOINT`，网络 Agent 需要 `NET_BGP_BOUNCE`。ASys 只负责给这些手术刀提供刀柄（鉴权与传输），不干涉刀刃（具体逻辑）。这一层提升了应用上限（High Ceiling）——领域专家可以在协议框架内构建任意深度的指令生态。

**结论**

瘦核心保证长期兼容性；胖扩展保证实际可用性。机制（Core）与策略（Standard ISA + Vendor Extensions）的分离，使 ASys 能够在保持协议稳定的同时，支持不同垂直领域的扩展需求。

## 3. 当前的系统操控选项

### 3.1 现有方案及其适用场景

当 AI Agent 需要操控系统时，目前有几类可用的方案。每种方案都有其优势、适合场景和团队背景。

**SSH + Shell 脚本 / Ansible**

这是最成熟、最通用的选择，30 年生产验证，几乎所有系统都原生支持。Ansible 的 agentless 架构是显著优势——目标机器不需要安装任何客户端，SSH 即可。YAML Playbook 对运维工程师可读性极高，团队协作、代码审查、审计都很自然。幂等性设计经过大规模实践验证，错误处理和 retry 逻辑成熟。

对于人类工程师主导、以配置管理和批量部署为主的场景，这套方案是合理的首选。当 AI Agent 通过 SSH 调用这些工具时，它实际上是在"模拟人类工程师的操作方式"——这不是批评，而是描述：这套工具就是为人类设计的，Agent 借用它是一种自然的选择。但这也意味着，非结构化的文本输出、缺乏协议级权限粒度，是这套方案在 Agent 场景下的固有局限。

**Kubernetes Operator / 声明式 API**

K8s Operator 代表了一种不同的哲学：声明期望状态，由控制器负责收敛。这种方式在云原生应用层表现出色——自动处理中间状态、失败重试、滚动更新，背靠整个 CNCF 生态，工具链和可观测性都是顶级的。

对于运行在 K8s 上的应用，Operator 是管理其生命周期的自然选择。它的适用边界是 K8s 管理的资源层；裸金属服务器、嵌入式设备、或 K8s 本身出问题时的底层恢复，不在它的设计范围内。

**自制 gRPC / 二进制协议 over SSH**

有些团队为 Agent 定制了自己的控制面：在 SSH 或 TLS 上封装 gRPC，用 Protobuf 定义强类型接口。这条路的优势是完全掌控协议细节，按需裁剪，性能好，类型安全。对于已有 gRPC 基础设施的团队，这是自然的选择。

gRPC 是通用 RPC 框架，它解决的是"数据怎么传"的问题。但在 Agent-to-OS 场景下，还有一层问题需要每个团队自行解决：操作指令的语义如何定义、权限边界如何在协议层表达、防重放和审计如何内置。这不是 gRPC 的职责范围，也意味着这部分工作无法跨团队复用。

**ASys**

ASys 在上面这些选项之外提供了另一条路：专门为 AI Agent 作为操作者设计，从帧格式到鉴权机制到审计链都以"Agent 是主要客户端"为前提。

它不是要取代上面任何一种方案。一个系统完全可以同时运行 Ansible（供人类工程师使用）和 ASys（供 AI Agent 使用）。它试图填补的空白是：当 Agent 需要高频、低延迟、可审计地操控系统，且需要在极端场景下保持接口可用时，目前没有一个专门为这个场景设计的标准化方案。

### 3.2 ASys 的设计取舍

选择 ASys 意味着接受一些具体的权衡：

| 设计选择 | 带来的好处 | 对应的代价 |
|---------|-----------|-----------|
| 静态内存预分配 | OOM 场景下进程存活，延迟确定性 | 并发连接数有上限（当前 8，可编译调整） |
| 零外部依赖（仅 C 标准库） | 任何 POSIX 环境可部署，无运行时风险 | 功能需自行实现，没有生态库可借用 |
| Capability Map 正向授权 | Agent 权限边界精确，无法越权 | 每个 Agent 需要管理员手动注册公钥 |
| 二进制 APDU 帧 | 解析零开销，格式不随系统版本漂移 | 调试不如文本协议直观，需要工具支持 |
| Noise IK 而非 mTLS | 无证书依赖，握手 1-RTT | 没有 PKI 生态，密钥管理需自行处理 |
| append-only 审计链 | 每条指令可追溯，不可篡改 | 存储开销，高频场景需要异步写入优化 |

这些不是"ASys 更好"的论据，而是"如果你的场景恰好需要这些属性，ASys 值得考虑"的依据。

### 3.3 适合考虑 ASys 的场景

基于上述取舍，ASys 在以下场景下有较好的适用性：

**需要 Agent 高频采样系统状态**：长连接 + 二进制帧，`SYS_STATUS` 的 RTT 在缓存命中时 < 1ms，适合 Agent 实时监控系统指标。

**极端场景下的最后接口**：SSH 可能在 OOM 时失联，`asyd` 设计上在这种情况下仍然存活（Core ISA 诊断指令始终可用；依赖 `fork()` 的操控指令在严重 OOM 下可能返回 `0x6F01`，Agent 可据此切换只读诊断路径）。

**需要精确控制 Agent 权限边界**：Capability Map 精确到单条指令，可以给不同 Agent 授予不同的指令子集，权限变更通过 SIGHUP 热重载生效。

**裸金属 / 嵌入式 / 无容器编排的环境**：零外部依赖，单一静态二进制，任何能跑 C 程序的 POSIX 环境都可以部署。



## 4. 协议机制概述

### 4.1 帧格式：站在巨人肩膀上

ASys 的帧格式借鉴自 APDU（Application Protocol Data Unit）指令体系——这是迄今为止部署规模最大的二进制指令协议之一。APDU 作为应用层指令格式，被 ISO/IEC 7816（接触式智能卡）、ISO 14443（非接触式卡）等多个国际标准采纳：每一张 EMV 银行芯片卡、每一本电子护照、每一张 NFC 交通卡和门禁卡、每一张 SIM 卡的应用层，运行的都是同一套 APDU 结构。仅全球 SIM 卡存量就超过 100 亿，叠加银行卡、证件类、门禁与交通卡，APDU 协议体系覆盖的设备数量在数百亿量级——它的每一个边界条件、每一个错误路径，都经过了极其严苛的实战验证。

APDU 的设计哲学是"指令 + 参数 + 期望响应长度"，本质上是为**机器操控机器**而生的：读卡器（机器）向智能卡（机器）发送结构化指令，从不经过人眼解析。这与 ASys 的核心命题——AI Agent 直接操控系统——在设计直觉上完全同构。其状态字（SW）体系已成行业惯例，`0x9000` = Success、`0x6982` = Access Denied。ASys 在此基础上重新定义了 CLA 字节的位域，并扩展了 `0x6Fxx` 子码体系（低字节携带内核错误子码），将这套经过数十年验证的帧结构移植到 Agent 时代。

默认端口 **7816** 是对 ISO/IEC 7816 这一技术渊源的显式致敬。

**标准帧布局（7 字段）：**

```
[CLA][INS][P1][P2][Lc][Data][Le]
  1    1    1   1   1   var   1   (字节)
```

| 字段 | 长度 | 语义 |
|------|------|------|
| `CLA` | 1B | 指令类别：协议版本（bit7-5）、链式标志（bit4）、安全级别（bit3-2）、扩展帧（bit1） |
| `INS` | 1B | 指令码，直接索引指令分组 |
| `P1/P2` | 各 1B | 指令参数 |
| `Lc` | 1B | Data 段字节数 |
| `Data` | 变长 | 业务数据（含 Seq 序列号） |
| `Le` | 1B | 期望响应长度；`0x00` 表示仅返回状态字 |

安全帧在 `Le` 之后附加 16 字节 Auth Tag（HMAC-BLAKE2b，见 §2.2.3），不计入 `Lc`，覆盖完整帧头 + Epoch_ID + Seq + Payload——任何字段被篡改均导致验签失败。

### 4.2 指令集分层：High Nibble Paging

ASys 以 INS 字节的**高位半字节（High Nibble）**作为逻辑页索引，将 256 个指令槽划分为四个层次：

| 编码范围 | 分组 | 治理规则 |
|----------|------|---------|
| `0x00-0x0F` | **Core ISA** | 协议宪法——已定义指令永久锁定，任何版本均不可更改 |
| `0x10-0x1F` | **Protocol Control** | 协议内部保留（`0x11` TASK_QUERY、`0x12` TASK_CANCEL 等） |
| `0x20-0x8F` | **Standard ISA** | 7 个功能组（Process / Network / Diagnostics / Storage / Security / Runtime / Hardware），语义锁定 |
| `0x90-0xBF` | **Standard ISA 预留** | 协议组保留，未来新增标准指令组 |
| `0xC0-0xFF` | **Vendor Extensions** | 通过 4 字节 `Vendor_ID` 支持无限厂商复用 |

这个设计让任何开发者看到指令码就能判断所属分组：`0x2x` 一眼知道是 Process Control，`0x3x` 是 Network Control，无需查表。

Core ISA 是 Agent 观察世界的"视网膜"——`0x00 SYS_CAPS` 返回节点能力位图和静态描述符，`0x01 SYS_HELLO` 建立会话与时钟对齐，`0x02 SYS_STATUS` 提供高频（10Hz+）的系统实时体征，`0x03 SYS_PROCS` 返回高负载进程摘要。这四条指令无副作用，握手完成后即可调用，是 Agent 感知节点状态的基础原语。

### 4.3 连接建立：三步走

从 TCP 连接到可发送业务指令，共经历三步：

**第一步：Client Magic（4 字节）**

TCP 连接建立后，Agent 先发送 4 字节 Magic（`0x41535953`，ASCII: "ASYS"）。Magic 不符或超时则立即断开，不推送任何数据，通用端口扫描器无法触发服务端响应。

**第二步：Pre-Handshake Frame（38 字节明文）**

Magic 验证通过后，`asyd` 推送：

```
[Magic(4B)][Version(2B)][ServerPubKey(32B)]
 0x41535953  0x0100      Curve25519 静态公钥
```

这一步解决了密钥分发问题——Agent 无需带外获取服务端公钥，`asyd` 在握手前主动告知。Agent 收到后与 `~/.asys/known_hosts` 比对指纹：首次连接等待管理员确认并写入，后续自动比对。公钥明文传输没有安全损失——安全性依赖的是指纹确认，与 SSH 首次连接的机制完全一致。

> **SDK 实现安全要求**：Pre-Handshake Frame 解决的是密钥传输问题，不解决首次连接的信任建立问题。SDK 在 `known_hosts` 无记录时**严禁自动写入未确认的指纹**（自动 TOFU）。自动信任意味着内网 ARP 欺骗或 DNS 污染即可完成中间人攻击。在动态弹出节点的云原生场景中，推荐通过编排系统（K8s Secret / Ansible / Vault）将节点公钥**预注入** `~/.asys/known_hosts`，在首次连接前完成信任建立。

**第三步：Noise IK 握手（1-RTT）**

```
Agent → asyd:  [e, es, s, ss]   ← Agent 临时公钥 + Agent 静态公钥（加密传输）
asyd  → Agent:  [e, ee, se]     ← asyd 临时公钥
```

握手完成后双方各持对称会话密钥。`asyd` 立即从握手密钥派生 `Epoch_ID`（4 字节，不在网络传输），用于后续所有指令的 Auth Tag 计算。同时检查 Agent 静态公钥是否在 `authorized_agents` 白名单，不在则返回 `0x6982` 断开。整个过程 1-RTT 完成。

### 4.4 一次完整交互

以 `SVC_RESTART`（异步指令，重启 nginx）为例，展示从握手后到结果确认的完整链路：

```
① Agent → asyd:  [CLA=0x04][INS=0x22][Lc=0x0A][Seq=42][nginx][Le=0x06][Auth Tag]
                   Sec=01（Signed）                 服务名不含 .service 后缀

② asyd  → Agent:  [Task_Handle=0x0001AABB][SW=0x9000]
                   入队成功，fork 已发起

③ Agent → asyd:  [INS=0x11][Lc=0x04][0x0001AABB][Le=0x03]
                   TASK_QUERY，3 秒后查询

④ asyd  → Agent:  [Status=0x01][SW=0x9000]
                   Success，Handle 释放
```

几个细节值得注意：`SVC_RESTART` 的服务名不含 `.service` 后缀——`asyd` 内部自动拼接，从协议层切断路径注入攻击面；`0x9000` 在第②步代表"入队成功"而非"执行完成"，Agent 必须通过 `TASK_QUERY` 轮询最终状态；Task_Handle 高 16 位绑定 Session_ID，查询不属于当前 Session 的 Handle 一律返回 `0xFF`，不泄露任何信息。

### 4.5 协议演进保证

ASys 对长期兼容性有明确的协议级约束：

- **偏移量不变性**：Core ISA 任何现有字段的字节偏移永远不变，新字段只能占用预留空间或在末尾扩展
- **原子化替代**：若某字段失效，填充哨兵值（`0xFF` / `0x0000`）而非缩短响应长度——同一版本内响应字节总长永远不变
- **静默忽略**：Agent 必须能安全忽略未知 Reserve 字段，服务端必须能忽略未知填充位

这三条准则共同保证：今天写的 Agent 解析代码，期望在 50 年后的某个 POSIX 兼容系统上仍然有效。

## 5. 安全模型

ASys 的安全架构分为四层，从传输到执行形成纵深防御：传输加密（Noise IK）→ 身份验证（公钥白名单）→ 权限隔离（Capability Map）→ 审计追溯（append-only 审计链）。每一层都是协议的物理属性，而非可配置的策略选项。

### 5.1 传输安全：为什么不用 mTLS

ASys 放弃了行业常见的 mTLS，选择 **Noise Protocol Pattern IK**（Curve25519 + ChaCha20-Poly1305 + BLAKE2b）。这不是偏好问题，而是设计约束决定的：

mTLS 依赖 PKI——CA、证书、吊销列表（CRL/OCSP）、证书轮换。`asyd` 的核心使命之一是在系统极端压力下（OOM、磁盘满）仍然存活；一个因证书过期而失联的急救接口，比没有接口更危险。此外，OpenSSL/BoringSSL 是重量级依赖，与 ASys 零外部依赖的设计哲学根本冲突。

Noise IK 的三个优势恰好对应这三个问题：**零证书依赖**（纯密码学原语，无需 CA）、**单文件实现**（基于 Monocypher，约 2000 行 C 代码，可完整审计）、**1-RTT 握手**（IK 模式要求 Initiator 预知服务端公钥，与 Pre-Handshake Frame 机制天然配合）。

Noise IK 还带来一个协议层的额外红利：握手完成后 Agent 静态公钥即为身份标识，可直接索引 Capability Map，无需任何额外的身份层。

**前向安全性说明**：Noise IK 的会话密钥由临时密钥（每次握手独立生成）参与派生，静态私钥泄露**不能解密历史会话内容**。真实风险边界是：服务端静态私钥泄露后，历史握手消息中被加密的 Agent 静态公钥（身份）理论上可被还原——即攻击者可知道哪些 Agent 连接过该节点，但无法获知它们的操作内容。后续版本计划引入 Noise 规范原生支持的 **Rekey 机制**，在长连接运行期间周期性更换会话密钥，进一步收窄静态密钥泄露的影响窗口。

### 5.2 身份模型：比 SSH 更简单

ASys 的身份模型刻意设计得比 SSH 更简单，没有用户名、没有密码、没有证书——只有公钥。

```
服务端（asyd）                    客户端（Agent）
/etc/asyd/
├── id_curve25519      ←私钥      ~/.asys/
├── id_curve25519.pub  ←公钥      ├── id_curve25519
└── authorized_agents  ←白名单    ├── id_curve25519.pub
                                  └── known_hosts
```

两个信任方向完全对称：

**Agent 信任节点**：首次连接时，`asyd` 通过 Pre-Handshake Frame 推送服务端公钥，Agent 显示指纹等待确认，写入 `~/.asys/known_hosts`；后续连接自动比对，指纹不符则拒绝——与 SSH `known_hosts` 机制完全一致。

**节点信任 Agent**：管理员将 Agent 公钥写入 `/etc/asyd/authorized_agents`，握手完成后 `asyd` 检查公钥是否在白名单，不在则返回 `0x6982` 断开——与 SSH `authorized_keys` 机制完全一致。

白名单支持 SIGHUP 热重载：`sudo kill -HUP $(pidof asyd)` 即可添加或吊销 Agent 身份，现有连接不受影响。**一台机器，一个身份**——克隆虚拟机时必须删除 `id_curve25519`，`asyd` 首次启动会自动生成新密钥对，防止多节点共享同一指纹导致 Capability Map 冲突。

### 5.3 权限隔离：Capability Map 与防侧信道

Noise 握手完成、身份确认后，`asyd` 根据 Agent 静态公钥查询对应的 **Capability Map**——一张精确到单条指令的二进制位图。

选择位图而非 ACL 规则表，有两个非直觉的原因：

**O(1) 查找**：`(bitmap >> ins) & 1` 是单条机器指令。面对大规模恶意探测时，`asyd` 可在极低 CPU 消耗下拒绝非法请求，不存在 ACL 规则表的 O(n) 线性扫描。

**防侧信道攻击**：若先查权限再查存在性，攻击者可通过响应时间的微小差异推断"存在但被禁用"的指令，从而反推系统能力边界。ASys 的分发拓扑强制**先查存在性、再查权限**——所有未授权操作的响应时间完全一致，消灭信息泄露面。

```c
// 存在性检查（O(1) 位图）
if (!is_instruction_in_bitmap(ctx->cap_map, pkt->ins))
    return SW_NOT_FOUND;    // 0x6A81：物理不存在

// 细粒度权限校验
if (!check_permission(ctx, pkt))
    return SW_ACCESS_DENIED; // 0x6982：越权，操作被拒绝
```

`0x6A81`（物理不存在）与 `0x6982`（越权，操作被拒绝）的语义差异至关重要：前者意味着"我违规了"，后者意味着"环境不允许"——Agent 据此触发不同的自愈逻辑。

### 5.4 防重放：Epoch_ID 的设计

ASys 采用 **Epoch_ID + 会话内递增 Seq** 的双层防重放机制，不依赖磁盘持久化。

`Epoch_ID` 是其中最值得说明的设计：

```
Epoch_ID = HMAC-BLAKE2b(recv_key, "asys-epoch-v1")[:4]
```

它从 Noise IK 握手密钥派生，**永远不在网络上传输**——双方独立计算，结果相同。每次握手产生唯一的 `Epoch_ID`，混入每条指令的 Auth Tag 计算中：

```
Auth_Tag = HMAC-BLAKE2b(recv_key, Header‖Epoch_ID‖Seq‖Payload)[:16]
```

这意味着截获旧会话的指令包，在新会话中重放时，`Epoch_ID` 不同导致 HMAC 验签直接失败（`0x6982`）——无需查询任何持久化状态，服务端崩溃重启后旧包自动失效。会话内重放则由内存中的 `last_seen_seq` 拦截（`0x6985`）。

Auth Tag 验签强制使用常量时间比较（Monocypher `crypto_verify16()`），严禁 `memcmp()`——后者在发现第一个不匹配字节时提前返回，攻击者可通过响应时间差逐字节推测 Auth Tag。

### 5.5 审计黑匣子

§1.2 中推导的第四个接口属性——**内建审计**——在这里落地。

每条 Standard ISA 及 Vendor Extension 指令在执行前，必须先写入审计记录：

```
[Timestamp(8B)][Agent_PubKey_Hash(4B)][INS(1B)][CLA(1B)][Seq(4B)][Param_Digest(4B)][SW(2B)]
合计：24 字节定长
```

定长二进制格式有两个刻意的设计：**防格式注入**（文本日志可被精心构造的参数内容污染，二进制定长完全免疫）；**可直接对接 SIEM**（无需解析器，`mmap` 后随机访问）。

审计日志通过文件系统属性强制只增（Linux：`chattr +a`），任何 Agent 指令无法修改已写入的记录，即使持有高权限。时序上**写前推送**：先落盘审计记录，再执行指令——即使执行失败，"曾经尝试过什么"永远可查。

当审计存储失效时，`asyd` 不静默继续运行，而是在每个响应帧上标记 `AUDIT_DEGRADED`——将降级状态透明暴露给 Agent，由 Agent 决定是否继续执行高风险指令。透明失败，而非静默失败。

> 注：审计黑匣子为设计目标，当前 v0.3.1 尚未实现，计划在后续版本落地。

## 6. 路线图与展望

### 6.1 当前状态

asyd v0.3.1 是 ASys 的第一个公开版本，对应协议版本 v1.0。两个版本号独立管理：协议版本在帧格式或指令语义发生变化时才 bump；软件版本随实现迭代独立演进。

协议 v1.0 覆盖的已稳定实现部分：Core ISA 四条观测指令（`SYS_CAPS` / `SYS_HELLO` / `SYS_STATUS` / `SYS_PROCS`）、首批 Standard ISA 操控指令（`PROC_THROTTLE` / `SVC_RESTART` / `TASK_QUERY`）、完整的 Noise IK 加密通道、公钥白名单身份验证、Epoch_ID 防重放、Pre-Handshake Frame 密钥分发机制。

OODA 闭环的各个环节已在 RHEL 上完成协议功能验证：服务端运行 CPU hog 触发负载异常，通过 Python 客户端依次调用 `SYS_STATUS` 发现异常、`SYS_PROCS` 定位进程、`PROC_THROTTLE` 压制、`SYS_STATUS` 核实恢复，各指令均返回 `0x9000`。

规范中已定义但尚未实现的部分（per-Agent 细粒度 Capability Map、审计黑匣子、链式传输、握手首包去重）标注为 planned，不计入协议 v1.0 的承诺范围，计划在后续版本逐步落地。

### 6.2 SDK 路线

协议本身解决的是"如何通信"的问题；SDK 解决的是"开发者如何用得上"的问题。两者同等重要。

**Python SDK**（近期）：面向 AI Agent 开发者，目标是能直接嵌入 LangChain、AutoGen 等主流 Agent 框架。支持 `asyncio` 异步 IO、自动重试、幂等保护，自动管理 `~/.asys/` 身份和 `known_hosts` 指纹验证。

SDK 的一个核心设计原则是：**Agent 永远不直接操作字节层**。Agent 调用的是结构化接口（如 `sdk.proc_throttle(pid=1234, action="stop")`），SDK 负责参数类型校验、边界检查，再将其编译为二进制帧。LLM 对字节偏移天然不敏感，让 Agent 直接拼接二进制参数是错误的使用方式；SDK 的强类型验证层是 ASys 安全模型在 Agent 侧的延伸。

### 6.3 协议演化

Standard ISA 目前定义了 9 条指令，`0x20-0x8F` 的 112 个槽位大部分仍是空白。这是刻意保留的——协议的长期有效性比早期的功能完整性更重要。新指令的添加会随实际场景需求驱动，而非预先规划。

Vendor Extensions（`0xC0-0xFF`）是协议开放性的核心机制。数据库、网络设备、云原生运行时等垂直领域，可以在 ASys 的鉴权与传输框架下定义专属指令集，无需修改协议核心。后续计划建立 Vendor_ID 注册规范和第三方扩展的标准化流程。

更远期的方向是引入基于 Wasm 的轻量插件沙箱，让第三方厂商在不修改 `asyd` 核心的情况下扩展指令集——前提是定义 ASys-WASI，将插件的能力边界与 Capability Map 对齐。这仍是研究方向，不是承诺。

### 6.4 一点想法

ASys 是一次尝试：如果为 AI Agent 从零设计一个系统接口，它应该是什么样的。

这个问题没有唯一答案。二进制帧、Noise IK、Capability Map——每一个选择都有其权衡，设计注记（`asys-design-notes.md`）记录了所有"为什么不用另一种方案"的推导过程，欢迎挑战。

如果 ASys 能引发一些关于"Agent 与系统之间的接口应该长什么样"的讨论，这个项目的目的就达到了。
